# Как писать description для скилла

Поле `description` в начале `SKILL.md` попадает в системный контекст агента. По нему агент решает, подходит ли скилл к запросу пользователя. От формулировки зависит, будет ли скилл применяться в нужных ситуациях.

## Рекомендации

### 1. Пишите в третьем лице

Description описывает скилл объективно, а не от лица «я» или «вы»:

- ✅ Хорошо: «Обрабатывает Excel-файлы и формирует отчёты»
- ❌ Плохо: «Я могу помочь обработать Excel»
- ❌ Плохо: «Вы можете использовать этот скилл для Excel»

### 2. Указывайте и ЧТО, и КОГДА

- **ЧТО** — какие действия выполняет скилл (конкретные возможности).
- **КОГДА** — в каких формулировках или ситуациях его применять (триггеры).

Пример:

```yaml
description: Извлекает текст и таблицы из PDF, заполняет формы, объединяет документы. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
```

### 3. Будьте конкретны

Чем точнее описание, тем реже агент применит скилл не к месту или пропустит нужный случай.

- ✅ Хорошо: «Генерирует коммит-сообщения по diff. Use when the user asks for help writing commit messages or reviewing staged changes.»
- ❌ Плохо: «Помогает с гитом»

## Примеры «хорошо / плохо»

| Плохо | Хорошо |
|-------|--------|
| «Помогает с документами» | «Извлекает текст и таблицы из PDF, заполняет формы. Use when working with PDF files or when the user mentions PDFs or document extraction.» |
| «Делает ревью кода» | «Ревью кода на качество, безопасность и лучшие практики по стандартам команды. Use when reviewing pull requests or when the user asks for a code review.» |
| «Настраивает проект» | «Создаёт правила Cursor для проекта. Use when the user wants to create a rule, add coding standards, or asks about .cursor/rules/ or AGENTS.md.» |
